# Feature: Environment Simplification

## Summary

Replace the current multi-file env system (global/local files per repo, symlinks, `environment/` directory, PortAllocator, `{service.PORT}` interpolation) with a single `env.toml` per workspace. The TOML file has per-service sections (repos and non-repo services). Iara reads `env.toml` and generates flat `.env` files into each repo worktree. User env vars are no longer injected into terminals â€” only delivered via generated `.env` files. Scripts still get env injected (shell `$VAR` expansion requires it). Iara-internal vars (`IARA_*`, settings overrides) remain injected as before. Ports are plain env vars in `env.toml`, with automatic allocation on project/workspace creation.

## Problem

The current env system has too many moving parts:

- `.env.{repo}.global` in a shared `environment/` directory
- `.env.{repo}.local` per workspace
- Symlinks to global files in each workspace
- `syncEnvSymlinks()` boot-time repair
- `mergeEnvForWorkspace()` merging global + local per repo
- Env injection into `pty.spawn` for terminals and scripts
- `PortAllocator` auto-assigning ports with 20-port spacing
- `{service.PORT}` interpolation syntax in `scripts.yaml`

The user deals with 3 file concepts (global, local, generated) when all they want is one place to manage env vars.

## Design

### File Structure

```
my-project/env.toml                        # main workspace (source of truth)
my-project/app/.env                         # generated for app repo
my-project/api/.env                         # generated for api repo
my-project/workspaces/feat/env.toml         # feat workspace (source of truth)
my-project/workspaces/feat/app/.env         # generated for app worktree
my-project/workspaces/feat/api/.env         # generated for api worktree
```

### env.toml Format

TOML with one section per service (repos and non-repo services like databases).

```toml
[db]
PORT = "5432"

[app]
NEXT_PUBLIC_API = "http://localhost:3001"
PORT = "3000"

[api]
DATABASE_URL = "postgres://localhost:5432/mydb"
PORT = "3001"
```

### Generated .env

Each repo gets its section written as a flat `.env`.

`app/.env`:

```env
# Generated by iara - do not edit. Source: env.toml
NEXT_PUBLIC_API=http://localhost:3001
PORT=3000
```

`api/.env`:

```env
# Generated by iara - do not edit. Source: env.toml
DATABASE_URL=postgres://localhost:5432/mydb
PORT=3001
```

### Port Allocation

Ports are auto-assigned once at creation time, then owned by the user.

**Project creation:** base port = `3000 + (project_index * 100)`, where `project_index` is the project's position in the list of all projects ordered by `createdAt`. Discovery generates `env.toml` with `PORT` values starting at the base, incrementing by 1 per repo.

**Workspace creation:** copy `env.toml` from main, then offset `PORT` keys (and keys ending in `_PORT`) by `20 * workspace_index` â€” but only for repo sections. Non-repo service sections (e.g., `[db]`, `[redis]`) keep their ports unchanged (databases/caches typically run on fixed well-known ports shared across workspaces).

Example:

```
Project 0, main:        app=3000, api=3001
Project 0, workspace 1: app=3020, api=3021
Project 0, workspace 2: app=3040, api=3041
Project 1, main:        app=3100, api=3101
```

After creation, the user can edit ports freely â€” the allocation is a one-time convenience.

### Workspace Creation

When a new workspace is created, `env.toml` is **copied** from the main workspace with port values offset. Each workspace is fully independent after creation.

## Requirements

### R1 â€” env.toml as Source of Truth

- **R1.1** Each workspace has one `env.toml` at its root directory.
- **R1.2** Format is TOML with one section per service: `[<service>]` containing key-value pairs. Sections can be repos or non-repo services.
- **R1.3** On workspace creation, `env.toml` is copied from the project root (main workspace) with port values offset (per R4.11). If no `env.toml` exists in main, an empty one is created.
- **R1.4** Validation: keys must match `^[A-Z][A-Z0-9_]*$`.
- **R1.5** Unknown sections (not a known repo name and not a service in `scripts.yaml`) are preserved in the file but ignored during generation and injection, with a warning logged.

### R2 â€” .env Generation

- **R2.1** Iara generates a flat `.env` file into each repo worktree in the workspace.
- **R2.2** Each repo's `.env` contains only the vars from its `[<repo>]` section. Non-repo service sections do not generate `.env` files (no worktree to write to).
- **R2.3** Generated `.env` includes a header comment: `# Generated by iara - do not edit. Source: env.toml`
- **R2.4** Generation triggers:
  - On server boot (for all workspaces)
  - On `env.toml` file change (watched by EnvWatcher)
  - On `env.write` / `env.delete` RPC calls
- **R2.5** Manual edits to generated `.env` are overwritten on next generation.

### R3 â€” Env Injection

- **R3.1** **Terminals:** remove user env var injection from `pty.spawn`. User env vars are delivered exclusively via generated `.env` files in repo worktrees.
- **R3.2** **Scripts:** env vars from `env.toml` are still injected into script processes. The handler reads the repo's `[<repo>]` section and passes it via `spawn({ env })`. This is required because shell variable expansion (`$PORT`) only works with process env, not `.env` files on disk.
- **R3.3** Non-repo services (e.g., `db`, `redis`) can have their own section in `env.toml` and get env injected the same way as repos.
- **R3.4** All existing iara-internal injection remains unchanged:
  - `IARA_WORKSPACE_ID`, `IARA_WORKSPACE_DIR`, `IARA_PROJECT_ID`, `IARA_PROJECT_DIR`
  - `IARA_GUARDRAILS`, `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`
  - `cleanEnv()` filtering of `IARA_*` and `ELECTRON_*` from base process env

### R4 â€” Remove Port System, Use PORT from env.toml

- **R4.1** Remove `PortAllocator` from orchestrator and server.
- **R4.2** Remove `interpolateEnv` and `interpolateCommands` from orchestrator.
- **R4.3** Remove `{service.PORT}` syntax support from `scripts.yaml` parsing.
- **R4.4** Remove service-level `port` field from `scripts.yaml` schema.
- **R4.5** Remove service-level `env` block from `scripts.yaml` â€” all env comes from `env.toml`.
- **R4.6** Services matching a repo name run in the repo's worktree. Non-repo services (e.g., `db`, `redis`) run at the workspace root. Both can have sections in `env.toml` for env injection and health checks.
- **R4.7** Health checks for long-running scripts (`dev`): `handlers/scripts.ts` reads the `PORT` var from the repo's section in `env.toml` and passes it to the supervisor. If `PORT` is defined, TCP health checks run against that port. If no `PORT`, the script is marked as `running` (no health check).
- **R4.8** `autoDetect` uses the same `PORT` for attach detection. Services without `PORT` are not auto-detected.
- **R4.9** Keep `isPinnedPort` and `sharedByPort` in supervisor. A port is "pinned" when the service section belongs to a non-repo service (not matching any repo name). Non-repo services (db, redis) share ports across workspaces â€” if the port is already in use, the supervisor attaches as healthy without spawning.
- **R4.10** Scripts use `$PORT` from the injected env in their commands â€” same value, one source:
  ```toml
  # env.toml
  [backend]
  PORT = "3001"
  DATABASE_URL = "postgres://..."
  ```
  ```yaml
  # scripts.yaml
  backend:
    essencial:
      dev: "uvicorn app.main:app --port $PORT --reload"
  ```
- **R4.11** Port allocation at creation time:
  - **Project creation:** base port = `3000 + (project_index * 100)`, where `project_index` is the project's position in the list of all projects ordered by `createdAt`. Repo sections get `PORT = base + repo_index`. Non-repo services get well-known ports from config files.
  - **Workspace creation:** copy `env.toml` from main, offset keys named `PORT` or ending in `_PORT` by `20 * workspace_index` (where workspace_index is the count of existing non-main workspaces + 1) â€” only for repo sections. Non-repo service sections keep their ports unchanged.
  - After creation, ports are user-editable. The allocation is a one-time convenience, not a runtime system.

### R5 â€” Remove Legacy Env System

- **R5.1** Remove `environment/` directory and global env files.
- **R5.2** Remove `.env.{repo}.global` and `.env.{repo}.local` file concepts.
- **R5.3** Remove `ensureGlobalSymlinks`, `syncEnvSymlinks`, and all symlink logic.
- **R5.4** Remove `getGlobalEnvPath`, `getLocalEnvPath`.
- **R5.5** Remove `mergeEnvForWorkspace` (replaced by reading `env.toml`).
- **R5.6** No automatic migration. Legacy env files are ignored if `env.toml` doesn't exist â€” the workspace starts with no env. Users re-enter vars through the UI. Legacy files are not deleted automatically.

### R6 â€” EnvWatcher

- **R6.1** Watch `env.toml` files across all workspaces (project root + `workspaces/*/env.toml`).
- **R6.2** On change, regenerate `.env` in all repo worktrees of that workspace.
- **R6.3** Debounce writes (150ms).
- **R6.4** On boot, generate all `.env` files for all workspaces.
- **R6.5** Suppress own writes â€” when iara writes `env.toml` via RPC, don't trigger a redundant regeneration cycle (the RPC handler already triggers generation).

### R7 â€” RPC API Changes

- **R7.1** `env.list` â€” returns env entries grouped by service. Result: `{ services: { name: string; entries: EnvEntry[] }[] }`.
- **R7.2** `env.write` â€” writes entries to the workspace's `env.toml`. Params: `{ workspaceId, services: { name: string, entries: EnvEntry[] }[] }`.
- **R7.3** `env.delete` â€” deletes the workspace's `env.toml`. Params: `{ workspaceId }`.
- **R7.4** `env:changed` push event â€” simplified to `{ workspaceId }`.

### R8 â€” UI Changes

- **R8.1** Env editor shows one section per service (repos and non-repo services from `scripts.yaml`).
- **R8.2** Remove merged preview mode â€” what you see is what gets written.
- **R8.3** Add/remove/edit operations write directly to `env.toml`.
- **R8.4** Remove any reference to "global" and "local" levels in the UI.
- **R8.5** Sections are auto-populated based on discovered repos and services from `scripts.yaml`.

### R9 â€” Contract Changes

- **R9.1** Remove `EnvRepoEntries` type.
- **R9.2** Keep `EnvEntry` type (`{ key: string; value: string }`).
- **R9.3** Add `EnvData` type: `{ services: { name: string; entries: EnvEntry[] }[] }`.
- **R9.4** Update RPC method signatures per R7.
- **R9.5** Remove `level` and `repo` from env RPC params.

### R10 â€” Unified Script + Env Discovery

Discovery currently only looks at build config file names and asks Claude to generate `scripts.yaml` with `env` blocks, `port` fields, and `{service.PORT}` references. Scripts and env are now generated by a single agent since they're tightly coupled â€” the agent needs to know services to assign ports, and ports to write commands.

- **R10.1** A single discovery agent explores repos openly â€” reads package.json scripts, docker-compose configs, Makefiles, pyproject.toml, etc. â€” to understand the project's build system, services, and dev server configuration. No rigid pattern matching.
- **R10.2** The agent generates two artifacts that work together:
  - **`scripts.yaml`** â€” no `env` blocks, no `port` fields, no `{service.PORT}` syntax. Commands use `$PORT` and other `$ENV_VAR` references. `dependsOn` captures service dependencies (e.g., api depends on db, frontend depends on api).
  - **`env.toml`** â€” env vars per service section wired to match the dependency graph: `PORT` values (repos use project base port + index, non-repo services use well-known ports), and cross-service env vars with concrete port values (e.g., `DATABASE_URL = "postgres://localhost:5432/mydb"` in `[api]` using db's port, `NEXT_PUBLIC_API = "http://localhost:3001"` in `[app]` using api's port). The agent understands dependencies and ensures env vars reference the correct ports across services.
- **R10.3** Discovery receives the existing `env.toml` and `scripts.yaml` content (if any) as context. Existing values take precedence â€” discovery doesn't overwrite user config.
- **R10.4** The agent supports a user prompt for targeted changes. Examples:
  - No prompt (initial discovery): full exploration, generate both files from scratch.
  - `"change api port to 4000"`: update `PORT` in `[api]` section of `env.toml`, update any cross-references (e.g., `API_URL` in other sections).
  - `"add REDIS_URL to backend"`: add the var to `[backend]` in `env.toml`.
  - `"add a storybook script to frontend"`: update `scripts.yaml` only.
- **R10.5** Update `DiscoveryResultSchema` to include both scripts and env data.
- **R10.6** Remove `port` and `env` fields from `ServiceDiscoverySchema`. Keep `dependsOn` â€” it's about script execution ordering, not env.
- **R10.7** `CLAUDE.md` generation remains a separate agent, triggered in parallel during project setup.
- **R10.8** Both agents (CLAUDE.md + scripts/env) auto-trigger on project creation, running in parallel.

### R11 â€” Dependencies

- **R11.1** Add a TOML parser/serializer to `apps/server`. Recommended: `smol-toml` (zero-dep, small, handles round-trip).

## Decisions

| #   | Decision                                                 | Rationale                                                                                                                  |
| --- | -------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| D1  | TOML format                                              | Per-repo sections need structure; TOML is clean for flat key-value with sections                                           |
| D2  | Per-repo sections only (no shared)                       | Simpler model â€” each repo owns its env completely                                                                          |
| D3  | Copy + port offset on workspace creation                 | Independence after creation, no sync. Port offset avoids collisions.                                                       |
| D4  | Port allocation at project creation                      | Base port = `3000 + (project_index * 100)` where index is by `createdAt` order. One-time convenience, user-editable after. |
| D5  | No user env injection for terminals                      | `.env` in repo is the delivery mechanism for terminals/Claude Code.                                                        |
| D6  | Scripts still get env injected                           | Shell `$VAR` expansion requires process env. Handler reads `env.toml`, passes to supervisor.                               |
| D7  | Iara-internal injection unchanged                        | `IARA_*`, `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`, `IARA_GUARDRAILS` â€” injected as before                                        |
| D8  | `PORT` in env.toml drives both scripts and health checks | One source for port value â€” scripts read `$PORT`, supervisor reads `PORT` for TCP health checks.                           |
| D9  | Non-repo services get sections in env.toml               | Same treatment as repos â€” `PORT` for health checks, env injection for `$VAR` expansion                                     |
| D10 | No automatic migration                                   | Simpler than mapping legacy multi-file structure; users re-enter vars once                                                 |
| D11 | `smol-toml` dependency                                   | Zero-dep, small, supports parse + stringify                                                                                |
